🗄️ Estructura de los SGBD y Modelado de Datos

Una guía ordenada: desde las capas de abstracción hasta el modelo Entidad-Relación, con ejemplos interactivos para entender Entidades, Atributos y Relaciones.

📚 1. Capas de abstracción en un SGBD

Los SGBD organizan los datos en tres niveles para lograr independencia lógica y física:

🌐 Externa (Vistas)

Muestra solo una parte de la base de datos a cada usuario. Oculta detalles internos.

Ej: Vista de clientes activos.

🧠 Conceptual (Lógica)

Representación global de los datos: entidades, relaciones, restricciones. Independiente del almacenamiento.

Tablas lógicas, tipos de datos.

⚙️ Interna (Física)

Cómo se guardan los datos en disco: índices, particiones, compresión. Optimización de rendimiento.

Archivos .ibd, páginas.

📝 2. El LOG (Registro de transacciones)

El log es un archivo crítico que garantiza atomicidad y durabilidad. Escribe antes de modificar los datos (Write-Ahead Logging). Permite UNDO/REDO tras fallos.

-- Ejemplo conceptual de un log -- [LSN:01] BEGIN TRAN T1 [LSN:02] UPDATE Cuentas SET saldo=1200 WHERE id=5 (old=500, new=1200) [LSN:03] COMMIT T1

🏷️ 3. Tipos de SGBD: Relacionales, No relacionales y Distribuidos

📊 Relacionales (RDBMS)

Basados en tablas, SQL y ACID. Ej: MySQL, PostgreSQL, Oracle, SQL Server.

Perfecto para integridad referencial y transacciones complejas.

🍃 No relacionales (NoSQL)

Documentos, clave-valor, columnas anchas o grafos. Escalamiento horizontal. Ej: MongoDB, Redis, Cassandra, Neo4j.

Flexibilidad y gran volumen.

🌍 Distribuidos

Datos repartidos en múltiples nodos. Pueden ser relacionales (CockroachDB, Spanner) o NoSQL (Cassandra). Alta disponibilidad.

Tolerancia a fallos, escalabilidad global.

🔄 5. Del Modelo Conceptual al Modelo Lógico (Relacional)

Conceptual (MER): independiente de tecnología. Lógico: traduce a tablas, claves primarias y foráneas, resuelve atributos multivalorados y relaciones M:N. Aún sin detalles físicos.

📌 MODELO CONCEPTUAL
Entidades: CLIENTE, PEDIDO, PRODUCTO
Relación M:N "Compra" con atributo "cantidad"
➡️ Transformación ⬅️
📌 MODELO LÓGICO (Tablas)
CLIENTE(id PK)
PEDIDO(id PK, id_cliente FK)
PRODUCTO(id PK)
DETALLE_PEDIDO(id_pedido FK, id_prod FK, cantidad)
-- Ejemplo de transformación lógica (SQL DDL) -- CREATE TABLE CLIENTE (id_cliente INT PRIMARY KEY, nombre VARCHAR(100)); CREATE TABLE PEDIDO (id_pedido INT PRIMARY KEY, fecha DATE, id_cliente INT, FOREIGN KEY (id_cliente) REFERENCES CLIENTE(id_cliente)); CREATE TABLE PRODUCTO (id_producto INT PRIMARY KEY, nombre VARCHAR(100)); CREATE TABLE DETALLE_PEDIDO (id_pedido INT, id_producto INT, cantidad INT, PRIMARY KEY(id_pedido, id_producto), FOREIGN KEY (id_pedido) REFERENCES PEDIDO(id_pedido), FOREIGN KEY (id_producto) REFERENCES PRODUCTO(id_producto));

🧩 6. Ejemplo guiado: Biblioteca Digital

Aplicamos los conceptos: identificamos entidades, atributos y relaciones, luego construimos el modelo lógico.

📖 Entidades

LIBRO, AUTOR, PRESTAMO

🏷️ Atributos clave

LIBRO(isbn PK, título, páginas); AUTOR(id_autor PK, nombre); PRESTAMO(fecha_prestamo, fecha_devolución)

🔗 Relaciones

AUTOR escribe LIBRO (M:N) ➜ tabla AUTOR_LIBRO; PRESTAMO referencia LECTOR y LIBRO (1:N).
-- Tablas resultantes (Modelo Lógico) -- CREATE TABLE LIBRO (isbn VARCHAR(13) PRIMARY KEY, titulo VARCHAR(200), paginas INT); CREATE TABLE AUTOR (id_autor INT PRIMARY KEY, nombre VARCHAR(100)); CREATE TABLE AUTOR_LIBRO (id_autor INT, isbn VARCHAR(13), PRIMARY KEY(id_autor, isbn), FOREIGN KEY (id_autor) REFERENCES AUTOR(id_autor), FOREIGN KEY (isbn) REFERENCES LIBRO(isbn)); CREATE TABLE LECTOR (id_lector INT PRIMARY KEY, nombre VARCHAR(100)); CREATE TABLE PRESTAMO (id_prestamo INT PRIMARY KEY, fecha DATE, id_lector INT, isbn VARCHAR(13), FOREIGN KEY (id_lector) REFERENCES LECTOR(id_lector), FOREIGN KEY (isbn) REFERENCES LIBRO(isbn));

📌 Proceso completo de diseño de BD (resumen ordenado)

  1. Análisis de requisitos → identificar entidades, atributos y relaciones del dominio.
  2. Modelo Conceptual (MER) → diagrama ER con entidades, atributos (simples, compuestos, multi, derivados) y relaciones (cardinalidades).
  3. Modelo Lógico (Relacional) → transformar a tablas: cada entidad fuerte → tabla; atributos multivalorados → tabla independiente; relaciones M:N → tabla de enlace; definir PK y FK.
  4. Normalización (opcional pero recomendada) → eliminar redundancias y anomalías.
  5. Modelo Físico → implementar en un SGBD específico (tipos concretos, índices, particiones).

⚡ Independencia de datos (ventaja de las capas)

Independencia lógica: cambios en el esquema conceptual (agregar un campo) no afectan vistas externas.
Independencia física: cambios en almacenamiento (índices, reorganización) no afectan esquemas lógicos ni externos.

✅ Ejemplo: Agregar el atributo "email" a la entidad CLIENTE no requiere modificar las aplicaciones que solo usan nombre y teléfono, gracias a las vistas externas.
